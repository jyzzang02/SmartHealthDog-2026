# 🚀 기여 가이드라인 (CONTRIBUTING)

저희 프로젝트에 관심을 가져주시고 기여해 주셔서 감사합니다!

일관성 있고 효율적인 협업을 위해 개발 시 따라야 할 기본적인 규칙과 워크플로우를 안내합니다.

---

## 1. 🛠️ 개발 워크플로우

저희 프로젝트는 **GitFlow** 기반의 워크플로우를 따릅니다. **`develop`** 브랜치가 주 개발 라인이며, 모든 새로운 기능 및 버그 수정은 이 브랜치를 기반으로 진행됩니다.

1.  작업 전, **Issue**에 해당 기능 또는 버그 수정 사항을 먼저 등록합니다.
2.  작업자는 최신 **`develop`** 브랜치에서 새로운 **기능/버그 브랜치**를 생성합니다.
3.  브랜치에서 작업을 수행하고 로컬 테스트를 완료합니다.
4.  작업이 완료되면 **Pull Request (PR)**를 생성하여 코드를 병합 요청합니다.
5.  코드 리뷰를 거쳐 승인(Approve)되고 자동화된 테스트를 통과하면 **`develop`** 브랜치로 병합(Merge)됩니다.

> ⚠️ **주의:** **`main`** 브랜치 (혹은 `master`)는 **배포(Production)** 전용 브랜치입니다. `main` 브랜치에 직접 커밋하거나 PR을 병합하는 것은 **엄격히 금지**됩니다.

---

## 2. 🌳 Git 브랜치 이름 규칙

브랜치 이름은 해당 브랜치에서 수행하는 작업의 **유형**과 **기능 명세서 코드(ID)**를 명확히 포함해야 합니다.

### 브랜치 이름 구조

$$\text{type/CODE-ID-description}$$

| 요소 | 설명 | 예시 |
| :--- | :--- | :--- |
| **type** | 작업의 종류를 나타내는 접두사 | `feat`, `bugfix`, `hotfix`, `docs` |
| **CODE-ID** | 기능 명세서의 고유 코드 | `AUTH-03`, `TASK-123` |
| **description** | 해당 작업에 대한 간결한 설명 | `로그인-폼-추가`, `css-수정` |

### 규칙 세부 사항

* **소문자 (Lowercase)**만 사용합니다.
* 단어 구분에는 **하이픈 (`-`)**을 사용합니다.
* **공백이나 기타 특수문자** 사용을 금지합니다.
* **명세서 ID**를 반드시 포함해야 합니다.

### 예시

| 작업 내용 | 브랜치 이름 |
| :--- | :--- |
| **Feature** (회원 가입 기능, 코드: `AUTH-03`) | `feat/AUTH-03-회원가입` |
| **Bugfix** (로그인 에러 수정, 코드: `AUTH-01`) | `bugfix/AUTH-01-토큰-에러-수정` |
| **Documentation** (README 파일 업데이트) | `docs/readme-업데이트` |

## 3. 💬 Git 커밋 메시지 규칙

커밋 메시지는 프로젝트의 변경 이력을 체계적으로 관리하기 위해 명확하고 일관성 있게 작성해야 합니다.

### 커밋 메시지 구조

$$\text{type: subject (50자 이내)}$$
$$\text{(본문: 필요 시 변경 상세 내용 및 이유 기재)}$$

### 커밋 유형 (type)

| 유형 | 설명 | 예시 |
| :--- | :--- | :--- |
| **`Start`** | 프로젝트 초기 설정 또는 시작 | `Start: 프로젝트 초기 환경 설정` |
| **`Feat`** | **새로운 기능** 추가 | `Feat: 사용자 프로필 페이지 추가` |
| **`Fix`** | **버그 수정** | `Fix: 로그인 시 토큰 만료 버그 수정` |
| **`Design`** | CSS 등 사용자 **UI 디자인 변경** | `Design: 버튼 hover 효과 개선` |
| **`Refactor`** | **코드 리팩토링** (기능 변경 없음) | `Refactor: 유틸리티 함수 분리 및 개선` |
| **`Settings`** | 환경 설정 파일 변경 | `Settings: 개발 환경 변수 파일 수정` |
| **`Comment`** | 필요한 **주석 추가 및 변경** | `Comment: 복잡한 로직에 주석 추가` |
| **`Dependency/Plugin`** | 종속성/플러그인 추가 | `Dependency/Plugin: Axios 라이브러리 추가` |
| **`Docs`** | **문서 수정** (코드 외, e.g., README) | `Docs: README.md에 설치 방법 추가` |
| **`Merge`** | 브랜치 병합 | `Merge: develop 브랜치 최신 내용 병합` |
| **`Deploy`** | 배포 관련 작업 | `Deploy: 스테이징 서버 배포 환경 설정` |
| **`Rename`** | 파일 또는 폴더 **이름 변경 및 이동** | `Rename: user-page 폴더명을 profile로 변경` |
| **`Remove`** | **파일 삭제** 작업만 수행 | `Remove: 사용하지 않는 레거시 파일 삭제` |
| **`Revert`** | 전 버전으로 **롤백** | `Revert: 이전 커밋으로 롤백` |

## 4. 📝 Pull Request (PR) 가이드라인

모든 코드는 PR을 통해서만 **`develop`** 브랜치로 병합됩니다.

1.  **PR 생성 전 준비:**
    * 커밋 내역을 깔끔하게 **정리(Squash/Rebase)**합니다.
    * `develop` 브랜치의 최신 변경 사항에 대해 **Rebase**를 수행하여 충돌을 미리 해결합니다.
    * 모든 **테스트를 통과**했는지 확인합니다.
2.  **PR 제목 및 본문 작성:**
    * 제목은 커밋 메시지 규칙을 따르거나, 변경 내용을 요약합니다.
    * 본문에는 **Issue 링크**를 포함하고, *어떤 문제*를 *어떻게* 해결했는지 상세히 설명합니다.
3.  **코드 리뷰 및 병합 조건:**
    * **최소 1명 이상의 승인(Approve)**이 필요합니다.
    * CI/CD를 통해 **모든 자동화 테스트가 통과**해야 합니다.
      - Integration tests, unit tests 등

---

## 5. 💡 기타 규칙

* **코딩 컨벤션:** 프로젝트 내부에 정의된 (예: Linter, Formatter) **코딩 스타일 및 컨벤션을 준수**합니다.
* **테스트:** 새로운 기능이나 버그 수정 시, 반드시 이에 상응하는 **테스트 코드를 추가**해야 합니다.
* **`.gitignore`:** 불필요한 파일 (로그, 임시 파일, 빌드 결과물 등)이 Git에 포함되지 않도록 관리합니다.
